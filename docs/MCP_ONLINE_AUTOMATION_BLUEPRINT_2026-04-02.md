# MCP Online Automation Blueprint (2026-04-02)

이 문서는 US Alpha Seeker 운영 환경에서 **MCP를 어디까지 붙이고, 어디는 붙이지 말아야 하는지**를 정리한 실행 가이드입니다.

목표:

1. 온라인 기반(GitHub/Vercel) 완전 자동화에 가까운 운영
2. 장애 감지/분석/리포팅 자동화
3. 실거래 안정성(주문 경로) 훼손 없이 확장

---

## 1) 핵심 원칙 (반드시 유지)

1. **주문 실행 경로는 Direct API 우선**  
   - Alpaca 주문/정정/취소는 기존 코드 경로 유지
   - MCP는 실행 경로가 아니라 **운영·관측 경로**에 배치

2. **MCP는 Ops Plane 중심으로 사용**  
   - 장애 분석, 로그 요약, 문서화, 보고 자동화
   - 자동 보정은 “비파괴 작업”만 자동 허용

3. **자동 보정은 단계적 권한**  
   - P0: 알림 + 제안만
   - P1: 안전한 자동 조치(재시도, 상태 동기화)
   - P2: 위험 조치(주문 취소/감축)는 수동 승인 게이트 필수

---

## 2) 현재 상태

- 연동 중:
  - Notion MCP
  - Google Drive MCP
- 운영 자동화는 이미 GitHub Actions + Vercel 중심으로 동작
- 텔레그램/노션 적재는 현재 코드에서 Direct API 경로가 안정적으로 존재

---

## 3) 추천 MCP 우선순위

## P0 (즉시 가치/낮은 리스크)

1. **GitHub MCP**
   - 용도: 워크플로우 실패 감지, 실패 런 링크 수집, 이슈 자동 생성
   - 효과: 장애 triage 속도 상승

2. **Vercel MCP**
   - 용도: 배포/런타임 로그 조회, 에러 재현 포인트 캡처
   - 효과: “웹앱 정상 + API 정상” 검증 자동화

3. **Notion MCP (고도화)**
   - 용도: Incident/Guard/HF Tracker 데이터 품질 점검 자동화
   - 효과: 운영 이력 가시성 및 회고 품질 향상

## P1 (확장 가치 큼, 가드 필요)

4. **Perplexity MCP (분석 보조)**
   - 용도: 장애 원인 후보 요약, 뉴스/매크로 보조 분석
   - 주의: 매수/매도 의사결정 자동 반영 금지(리서치 보조 전용)

5. **Telegram MCP (Ops 제어 보조)**
   - 용도: 상태 조회/리포트 전송/운영 명령 인터페이스
   - 주의: 실주문 트리거는 MCP 경유 금지 (읽기/승인 워크플로우 중심)

## P2 (도입 전 설계 선행)

6. **Error/Observability MCP (Sentry/로그 허브 계열)**
   - 용도: 에러 집계, 알림 라우팅, 재발 패턴 분석
   - 효과: HF/Sidecar/웹 런타임 이상 탐지 자동화 고도화

---

## 4) 아키텍처 권장안 (2-Plane)

### A. Trade Plane (보수적, 고정)

- Sidecar dry-run / market-guard / 실주문 로직
- Direct API: Alpaca, Telegram, Notion Sync
- 원칙: 단순·예측 가능·최소 의존성

### B. Ops Plane (확장, MCP 중심)

- 장애 수집/요약/보고
- 로그 진단, 릴리즈 헬스체크
- 월간 아카이브 리포트 및 운영 품질 지표

---

## 5) “온라인에서 동작” 기준 정리

사용 기준:

1. GitHub Actions에서 실행 가능해야 함
2. Vercel 또는 외부 SaaS API로 접근 가능해야 함
3. 로컬 전용 stdio 의존 MCP는 운영 경로에서 제외

실무 권장:

- 운영 핵심은 API/Webhook 기반으로 구성
- MCP는 진단·분석·워크오더 생성 계층에 배치

---

## 6) 자동 보정(Autocorrect) 권장 범위

자동 허용(안전):

- transient 네트워크 실패 재시도
- Notion 동기화 재시도
- 아티팩트 누락 시 재생성 시도
- 요약 리포트 보강

수동 승인 필수(위험):

- 주문 감축/청산 강제 수행
- 정책 값 실시간 변경
- 실주문 경로 토글 변경

---

## 7) 추가로 있으면 좋은 기능 (MCP 무관 공통 강화)

1. **Health Gate**
   - “수집 정상/지표 정상/알림 정상” 체크 후 다음 단계 실행

2. **Runbook Auto-Link**
   - 실패 클래스별로 고정된 복구 문서 링크 자동 첨부

3. **Error Fingerprint**
   - 동일 에러 묶음 처리(중복 알림 억제)

4. **Cost Guard**
   - Perplexity/HF 호출량, 토큰 비용 임계치 경고

---

## 8) 실행 순서 (권장)

### Step 1 (이번 주)

- GitHub MCP + Vercel MCP 기반 장애 진단 루틴 고정
- Notion Incident Log 자동 보강(실패 클래스/재현 링크/처리 상태)

### Step 2 (다음 주)

- Perplexity MCP “보조 분석 전용” 연결
- 텔레그램에 “진단 요약 + 다음 액션” 표준 메시지 템플릿 적용

### Step 3 (안정화 후)

- 월간 보관 반자동(A안) → 완전 자동(B안) 전환 검토
- Error/Observability MCP 연동으로 재발 관리 자동화

---

## 9) 운영 결론

지금 단계에서 가장 좋은 방향:

- **실거래 경로는 Direct API 유지**
- **MCP는 Ops 자동화에 집중**
- **GitHub/Vercel/Notion 중심 P0부터 고정**

이 방식이 리스크를 가장 낮추면서도 확장성과 자동화 수준을 동시에 높입니다.
